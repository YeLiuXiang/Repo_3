# GitHub AI 项目管理 Agent 系统 v2

基于文档驱动的 GitHub 项目管理 Agent 系统，5 个角色 × 2 个 Agent = 10 个 Agent，覆盖从需求到发布的完整流程。

---

## 设计理念

**文档即交接**：每个 Agent 的输出是一个 Markdown 文档（`prd.md` → `plan.md` → `tasks.md` → Issues），后一个 Agent 读取前一个 Agent 的输出作为输入，无需显式的"交接机制"。

**人工确认节点**：关键操作（创建 GitHub Issues、写入 tasks.md）前需用户确认，避免误操作。

**只读分析分离**：`arch.analyze` 严格只读，`qa.bugreport` 创建 Issue 前展示草稿确认——分析与写入明确分离。

## 目录结构

```
github_private_v2/
├── v2.md                               ← 本文件，系统概览
├── templates/
│   ├── prd-template.md                 ← PRD 文档模板
│   ├── plan-template.md                ← 技术方案模板
│   └── tasks-template.md               ← 任务列表模板
└── .github/
    └── agents/
        ├── pm.prd.agent.md             ← 自然语言 → prd.md
        ├── pm.clarify.agent.md         ← 交互式澄清 prd.md
        ├── arch.plan.agent.md          ← prd.md → plan.md
        ├── arch.analyze.agent.md       ← 只读一致性分析（可选）
        ├── proj.tasks.agent.md         ← plan.md → tasks.md
        ├── proj.taskstoissues.agent.md ← tasks.md → GitHub Issues
        ├── dev.implement.agent.md      ← Issue + plan.md → 代码 + PR
        ├── dev.review.agent.md         ← PR Diff → 代码审查报告
        ├── qa.testplan.agent.md        ← AC → 测试用例 + testplan.md
        └── qa.bugreport.agent.md       ← 测试失败 → Bug Issue
```

## 5 角色 × 10 Agent

### PM（产品经理）

| Agent | 输入 | 输出 | 说明 |
|---|---|---|---|
| `pm.prd` | 自然语言需求描述 | `specs/<功能>/prd.md` | 生成完整PRD，含 AC、用户故事、范围 |
| `pm.clarify` | `prd.md` | 更新后的 `prd.md` | 逐条澄清模糊点，一次一问 |

### Architect（架构师）

| Agent | 输入 | 输出 | 说明 |
|---|---|---|---|
| `arch.plan` | `prd.md` | `specs/<功能>/plan.md` | 技术方案，含架构图、接口合约、ADR |
| `arch.analyze` | `prd.md` + `plan.md` | 控制台报告（只读） | 一致性检查，不修改任何文件 |

### Project Manager（项目经理）

| Agent | 输入 | 输出 | 说明 |
|---|---|---|---|
| `proj.tasks` | `plan.md` + `prd.md` | `specs/<功能>/tasks.md` | 生成任务列表，用户确认后写入 |
| `proj.taskstoissues` | `tasks.md` | GitHub Issues + Project Board | 批量创建 Issue，配置看板，写 Wiki |

### Dev（开发工程师）

| Agent | 输入 | 输出 | 说明 |
|---|---|---|---|
| `dev.implement` | GitHub Issue + `plan.md` | 代码 + PR | 按分支规范实现，PR 含 DoD 清单 |
| `dev.review` | PR Diff + `plan.md` | 代码审查报告 | 5维度审查，Blocker 含前后代码对比 |

### QA（质量工程师）

| Agent | 输入 | 输出 | 说明 |
|---|---|---|---|
| `qa.testplan` | `prd.md`（AC） | `specs/<功能>/testplan.md` | 4类测试用例 + AC覆盖矩阵 |
| `qa.bugreport` | 测试失败信息 | GitHub Bug Issue | P0/P1自动触发 PR Request Changes |

## 标准工作流

```
用户输入需求
    │
    ▼
[pm.prd] → specs/<功能>/prd.md
    │
    ├─ (可选) [pm.clarify] → 澄清 prd.md 模糊项
    │
    ▼
[arch.plan] → specs/<功能>/plan.md
    │
    ├─ (可选) [arch.analyze] → 一致性检查报告（只读）
    │
    ▼
[proj.tasks] → specs/<功能>/tasks.md  ← 用户确认后写入
    │
    ▼
[proj.taskstoissues] → GitHub Issues + Project Board + Wiki  ← 用户确认后创建
    │
    ├─ 并行：[qa.testplan] → specs/<功能>/testplan.md
    │
    ├─ 开发循环：
    │     [dev.implement] → 代码 + PR
    │     [dev.review] → 代码审查
    │
    └─ 测试循环：
          执行 testplan.md 中的测试用例
          失败时：[qa.bugreport] → Bug Issue（P0/P1 触发 Request Changes）
```

## 生成文档结构

运行系统后，`specs/` 目录会生成：

```
specs/
└── <功能目录名>/
    ├── prd.md        ← PM 产出
    ├── plan.md       ← Architect 产出
    ├── tasks.md      ← Project Manager 产出
    └── testplan.md   ← QA 产出
```

GitHub 上会生成：
- Issues：`[TASK] T-001: 描述`、`[BUG-P1] 描述`
- Project Board：Status / Priority / Sprint 字段
- Wiki：`PRD-<功能>`、`Plan-<功能>`

## 快速使用

1. **开始新功能**：启动 `pm.prd`，输入功能描述
2. **需要澄清需求**：启动 `pm.clarify`，指定 prd.md 路径
3. **生成技术方案**：启动 `arch.plan`，指定 prd.md 路径
4. **检查文档一致性**：启动 `arch.analyze`（可选，只读）
5. **分解任务**：启动 `proj.tasks`，指定 plan.md 路径
6. **创建 GitHub Issues**：启动 `proj.taskstoissues`，指定 tasks.md 路径
7. **编写测试计划**：启动 `qa.testplan`，指定功能目录
8. **实现功能**：启动 `dev.implement`，指定 Issue 编号
9. **代码审查**：启动 `dev.review`，指定 PR 编号
10. **提交 Bug**：启动 `qa.bugreport`，描述测试失败信息
